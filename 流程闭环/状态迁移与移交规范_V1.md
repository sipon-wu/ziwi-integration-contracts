# 流程闭环与状态移交规范 V1

> 核心原则：**每个流程要么闭环（最终关闭），要么把状态明确移交给下一流程。**
> 不允许任何工作项停在中间态无人认领、无判据、无时限。
> 每个状态都有四要素：**进入判据 / 退出判据 / 责任人 / 超时升级**。

---

## 〇、通用规则（所有流程共用）

1. **状态必须可见**：每个工作项（缺陷/部署变更/契约问题）在其所在文件或 issue 中标注当前状态，不靠口头。
2. **移交必须带"移交物"**：交出去时附证据（截图/commit/日志），接收方凭证据判进入判据。
3. **超时即升级**：停在任一非终态超过约定时限（默认 2 工作日），自动升级给上一级（开发→技术负责人→项目经理）。
4. **终态唯一**：`Closed` / `Released` / `已闭环` 是唯一结束态；任何其他状态都视为"未完成"。
5. **回退要记录**：任何从后向前的状态回退，必须在对应文件写一行原因。

## 一、提测 → QA → 上线 流水线（Stage-Gate）

适用：每次功能/修复进入 staging 前的全流程。

```
[开发中 Draft]
   │ 开发自测通过 GATE-01~08（见用例集第一章）
   ▼
[待QA Ready-for-QA]  ──移交物：自测记录 + commit 号
   │ QA 认领，开始执行用例
   ▼
[测试中 In-QA]
   │ 全部 P0 通过、P1≤5 个
   ▼
[可上线 QA-Passed]  ──移交物：QA 报告（通过用例数/提单号）
   │ 运维/发布负责人确认
   ▼
[已发布 Released]   ──终态
```

**回退路径**：
- `Ready-for-QA` 自检未过 → 退回 `Draft`，不退 QA 工时。
- `In-QA` 发现 P0 FAIL → 状态置 `QA-Failed` → 退回 `Draft`（开发修）→ 重新走 GATE。
- 上线后回归发现 P0 → `Reopened` → 回到 `Draft`。

| 状态 | 进入判据 | 退出判据 | 责任人 |
|------|---------|---------|--------|
| Draft | 需求明确/缺陷确认 | GATE-01~08 全过 | 开发 |
| Ready-for-QA | GATE 全过 + 自测记录 | QA 认领 | 开发→QA（移交） |
| In-QA | QA 认领 | 全部 P0 过、P1≤5 | QA |
| QA-Passed | 用例达标 | 发布确认 | QA→发布（移交） |
| Released | 发布完成 | 终态 | 发布负责人 |
| QA-Failed / Reopened | P0 失败/上线回归失败 | 修复重新 GATE | 开发 |

## 二、缺陷生命周期（Bug Lifecycle — 闭环）

每条用例 FAIL 或线上问题，都开一条缺陷，走完整生命周期，直到 `Closed`。

```
[新建 New] → [已分诊 Triaged] → [修复中 Fixing]
   → [待复测 Fixed] → [复测中 Verifying] → [已关闭 Closed]
                                      ↑            │
                                      └─ 复测 FAIL ┘
                                         [重新打开 Reopened] → Fixing
```

| 状态 | 进入判据 | 退出判据 | 责任人 |
|------|---------|---------|--------|
| New | 发现 FAIL/异常 | 确认复现 + 定级(P0/P1/P2) | 发现者 |
| Triaged | 已定级+指派 | 开发开始修 | QA/技术负责人 |
| Fixing | 已指派 | 代码改完 + commit | 开发 |
| Fixed | 有 commit + 自测 | QA 复测通过 | 开发→QA（移交） |
| Verifying | QA 认领复测 | 原用例转 ✅ | QA |
| Closed | 复测 ✅ + 关联用例更新 | 终态 | QA |
| Reopened | 复测 FAIL/回归 FAIL | 重新 Fixing | QA→开发 |

**闭环证据**：`Closed` 必须带来用例集对应行的"现状"从 ❌ 改为 ✅，否则不算闭环。

## 三、部署变更生命周期（各方互盯闭环）— 对应"部署日志 A"

适用：任何触碰 **nginx / SSL / 域名分流 / 反向代理** 的部署改动，全员平权共写。

```
[拟部署 Draft] → [已部署 Deployed] → [受影响方确认 Affected-Confirmed] → [已闭环 Closed]
                     │                                              │
                     └── 异常/踩到他人 ──▶ [回滚 Rollback] → Draft
```

| 状态 | 进入判据 | 退出判据 | 责任人 |
|------|---------|---------|--------|
| Draft | 计划改动，登记到 `deploy-log/` | 实际部署完成 | 改动方 |
| Deployed | 配置生效 + 自验通过 | 所有受影响方确认无踩踏 | 改动方 |
| Affected-Confirmed | 受影响团队（如 school/ecms）签字确认 | 超时未异议 → 自动 Closed | 各方（平权） |
| Closed | 终态 | — | 改动方 |
| Rollback | 出现跨服务故障 | 回滚完成 + 重新 Draft | 改动方 |

**登记模板**（`deploy-log/` 下每个文件追加一行，只追加不改历史）：

```
| 日期 | 团队 | 改了啥 | 影响范围 | 类型(nginx/SSL/域名) | 状态 | 确认方 |
| 2026-07-10 | mfg | cloud vhost 加 /api/v1/heartbeat 反代 | cloud+heartbeat 接入方 | nginx | Deployed→Affected-Confirmed | school/ecms |
```

**关键**：nginx/SSL 改动**必须**等受影响方确认才 Closed，这是"互盯"的强制闭环。

## 四、契约 / 接入问题生命周期（对应"契约 B"）

适用：`contracts/` 下接口契约的疑问、冲突、变更。

```
[提出 Open] → [已答复 Answered(mfg)] → [已确认 Confirmed] → [已闭环 Closed]
```

| 状态 | 进入判据 | 退出判据 | 责任人 |
|------|---------|---------|--------|
| Open | 接入方提问/冲突 | mfg 给结论 | 接入方→mfg |
| Answered | mfg 答复 + 改契约 | 接入方验证可行 | mfg |
| Confirmed | 接入方验证通过 | 契约定稿 | 接入方 |
| Closed | 终态 | — | 任一方 |

## 五、三套生命周期的共用看板（建议）

在共享仓根目录维护一个 `STATUS.md`，每行一个进行中的工作项：

```
| 类型 | 编号 | 标题 | 当前状态 | 责任人 | 超时日 |
| 缺陷 | BUG-001 | 班级切换不显数学 | Fixed(待复测) | 阿全 | 2026-07-12 |
| 部署 | DEP-007 | school1 加 443 | Affected-Confirmed | 技哥 | — |
| 契约 | CON-003 | heartbeat 频率未定 | Open | mfg | 2026-07-11 |
```

> 看板不是额外负担：它就是把"谁在哪儿卡着"显形，逼每条流程闭环或移交。

## 六、测试左移：Agent 三阶段前置（对接约束指南）

上面第一章的 `Draft` 状态**不是"开发闷头写代码"**，而是测试 Agent 必须左移介入的窗口。依据 `../测试标准/AI测试Agent约束指南.md`：

| 指南阶段 | 对应本状态机 | 产物（移交物） | 不完成的后果 |
|---------|------------|--------------|------------|
| 阶段一：需求评审 | `Draft` 起手 | 《质量风险分析》 | 无风险分析 → GATE 不退 QA |
| 阶段二：技术设计/代码开发 | `Draft` 进行中 | 《测试策略草案》+ diff 高风险点 | 无策略 → 提测被打回 |
| 阶段三：提测执行 | `Ready-for-QA`→`In-QA` | 《测试用例集》（见 QA_从严回测用例集_V1） | 直接跑用例，遗漏平台专项 |

**强制前置规则**：进入 `Ready-for-QA` 前，Agent 必须已加载 `AI测试Agent约束指南.md` 的**第七节平台专项检查单 + 第八节历史缺陷模式库**，并在提测说明里逐条标注覆盖情况（不涉及须显式说明）。这等同于给 GATE-01~08 加了"平台专项"维度，缺一不退 QA。

**阶段门串联**：
```
[需求评审 阶段一] ──《风险分析》──▶ [设计/开发 阶段二] ──《策略草案》──▶ [Draft 完成]
                                                                          │
                                           GATE-01~08 + 平台专项检查单全过 ▼
                                                              [Ready-for-QA] → 第一章流程
```
若前序阶段未参与，Agent 必须**回退**先做阶段一/二分析，不得直接进入执行——这是与第一章状态机的硬约束。
