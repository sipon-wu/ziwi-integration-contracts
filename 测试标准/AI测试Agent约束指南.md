# AI 测试 Agent 全流程参与约束工程指南（知微项目适配版）

> 原始指南为通用的「测试 Agent 左移 + 强制检查清单」方法论。本文在其后追加两节：
> - **第七节：知微项目适配**——把通用 10 条平台检查单映射到本项目（多租户 tenant/school 模型、RS256 统一登录、nginx 域名分流部署）。
> - **第八节：历史缺陷模式库**——已知逃逸点，作为下次类似需求的 Few-shot 自动提醒。
> 配套文件：`../流程闭环/状态迁移与移交规范_V1.md`（状态机）、`QA_从严回测用例集_V1.md`（全量用例）。

---

## 一、根本原因与解决思路

【症状】
- 只在"版本提交测试"时收到一句简单指令（如"测一下这个版本"）。
- Agent 缺乏需求背景、设计意图、代码变更影响面。
- 遗漏边界条件、异常流程、多租户/权限/兼容性等平台型产品特有细节。

【解药】
1. 流程约束：要求 Agent 必须在需求、设计、开发阶段就分步介入。
2. 上下文注入：每次对话必须提供结构化需求/设计/代码变更摘要。
3. 强制自检：让 Agent 在输出测试用例前，必须遍历内置的"平台细节检查清单"。
4. 产物标准：要求输出可视化的"测试分析与用例矩阵"，而非直接"跑一跑"。

---

## 二、流程约束：Agent 参与时间表

定义三个阶段，Agent 必须按阶段执行，未完成不得进入下一阶段：

【阶段一：需求评审期】（必须执行）
- 输入：产品需求文档(PRD) 或 用户故事。
- Agent 任务：生成《质量风险分析》，包含：多租户影响、上下游影响、可测试性要求。
- 输出模板："我识别到以下风险：... 为可测性，建议开发补充... 请确认。"

【阶段二：技术设计/代码开发期】（必须执行）
- 输入：技术方案、接口定义(OpenAPI/Swagger)、数据库变更DDL、核心代码diff。
- Agent 任务：生成《测试策略草案》，含接口级/数据级/业务级。
- 特别要求：必须分析代码 diff，识别未覆盖的分支逻辑，输出"高风险代码片段及建议测试点"。

【阶段三：提测执行期】（当前状态，需强化）
- 输入：阶段一、二的产物 + 提测邮件(版本号、环境、配置变更、灰度策略)。
- Agent 任务：基于前序分析，生成最终《测试用例集》，划分自动/手动。
- 若前序阶段未参与，Agent 必须主动要求补充上下文，并先回退执行阶段一/二分析，不得直接进入执行。

## 三、结构化提示词模板

所有提示词必须包含四要素：角色设定、强制上下文、平台细节检查清单、输出格式。

【角色】你是一个 10 年经验的大厂平台型产品测试架构师，极度关注多租户隔离、数据一致性、API 兼容性。
【上下文】本次需求为：{PRD}；涉及模块：{模块}；当前已知依赖：{上下游}。
【强制检查清单】在分析风险时，必须逐条回答（不涉及则标"不涉及"）：
1. 是否涉及多租户数据读写？哪个租户维度？隔离级别？
2. 是否修改了开放 API？请求/响应结构、签名方式、错误码是否有变？
3. 是否涉及消息发送/消费？消息格式、幂等处理、死信队列？
4. 是否涉及缓存/数据库变更？一致性策略是什么？
5. 是否包含异步回调、定时任务？失败补偿机制？
6. 是否有新的权限控制点？角色越权风险？
7. 是否需数据迁移？历史数据兼容性？
8. 灰度放量方式？开关配置及回滚方案？
9. 对于资金、资产、核心计数是否涉及？需要精确对账？
10. 弱网、超时、重试、重复请求场景是否考虑？
【输出格式】按"一、质量风险与影响面分析；二、可测试性要求；三、建议测试策略重点"结构化输出。

提测执行期强化提示词另含【强制细节清单】：设计用例时必须覆盖以下平台产品专项，每条至少 1 个用例：
1. 多租户数据隔离；2. 租户配额边界；3. API 鉴权异常（错误 AK/SK、过期时间戳、签名错误、重放）；4. API 版本兼容；5. 消息幂等；6. 分布式一致性；7. 缓存数据库双写；8. 灰度规则精准命中；9. 异常中断恢复；10. 页面/接口数据脱敏。
并输出"高风险场景补充用例"，针对代码变更的边界与异常逻辑。

## 四、如何让 Agent 主动查缺补漏（追问机制）

提示词末尾添加【自查与追问要求】："在你给出最终测试用例前，先反问自己：
- 我是否假设了所有依赖服务都正常？如果它们超时、返回错误码怎么办？
- 我是否考虑了租户 A 与租户 B 的数据完全隔离？有无共享缓存或文件存储？
- 如果这是线上第一次灰度发布，最坏可能会出现什么业务脏数据？
- 是否有新增或修改的定时任务？跑批期间对线上有多大压力？
- 我设计的用例能否被自动化为回归脚本？缺少什么测试数据？
如果上述任何问题你无法从现有上下文确认，必须明确列出待确认项并停止，要求我补充。"

## 五、持久化记忆与知识库约束

若平台支持知识库或长期记忆，必须植入：
- 平台架构文档摘要（服务拓扑、租户模型、关键数据流）。
- 历史线上故障库，要求 Agent 每次设计用例前检索相似故障模式。
- 通用平台测试检查清单（上文 10 条），作为每次任务的强制前置规则。

会话开始时可发送："请先加载《平台通用测试缺陷检查单》和《历史缺陷模式库》，本次测试必须规避已知高频逃逸点：多租户串数据、API 未鉴权、消息消费幂等、缓存不一致。所有用例必须体现对这些点的覆盖。"

---

## 六、度量与反馈闭环

每次线上发现逃逸缺陷，必须回溯到 Agent 的哪个阶段遗漏了，然后：
1. 将遗漏场景补充进检查清单。
2. 将该缺陷作为 Few-shot 示例，下次类似需求自动提醒 Agent。
例如追加提示词："上次上线曾发生因疏忽验证'租户配额耗尽后降级文案'导致投诉，本次若有配额相关变更，必须设计至少 2 条负向用例。"

## 七、知微项目适配（平台专项检查单映射）

本项目 = 知微 AI 教学助手（school 前端 + Go 后端，统一登录走 cloud 的 RS256 JWKS 验签，多租户维度 `tenant_id`/`school_id`/`campus_id`，部署为 nginx 域名分流 staging/prod 双环境）。

将通用 10 条检查单映射为**本项目强制前置**：

| # | 通用检查项 | 知微适配结论 | 优先级 |
|---|-----------|-------------|:------:|
| 1 | 多租户数据隔离 | **强适用 P0**：school/tenant/campus 三维隔离，跨校数据不可见；统一登录后 cloud 的 `products[]`/`tenant_id` 需正确注入 school 会话 | P0 |
| 2 | 租户配额边界 | 暂不涉及硬配额；但**班主任一人多班限制（A4 前提）**等价"业务配额边界"，需负向用例 | P1 |
| 3 | API 鉴权异常 | **强适用 P0**：cloud RS256 验签（错误/过期 token、重放 `iat`）、school HS256 会话、无 token 401 | P0 |
| 4 | API 版本兼容 | 部分适用：前端 `base` 路径、API `/api/v1` 版本、旧客户端降级提示 | P2 |
| 5 | 消息幂等 | **暂不适用**（当前无 MQ）；数据初始化 `upsert` 幂等等价，需验证重复导入不重复建班 | P1 |
| 6 | 分布式一致性 | 单体 + PostgreSQL 事务；**强一致要求**：班级归档/转班的数据一致性、upsert 冲突回滚 | P1 |
| 7 | 缓存数据库双写 | **适用**：redis 缓存（小微/学情），需验证更新后缓存脏数据 | P1 |
| 8 | 灰度规则精准命中 | **适用**：staging(`school1.ziwi.cn`) 先行验证，再发 prod；配置变更需回滚方案 | P0（发布门） |
| 9 | 异常中断恢复 | **适用**：服务重启/网络中断后状态恢复（自动保存草稿、Token 刷新） | P1 |
| 10 | 数据脱敏 | **适用 P0**：学生姓名/家长手机号/人脸等敏感信息在列表与导出中掩码 | P0 |

**强制前置规则**：每次提测，Agent 必须先加载本检查单 + 第八节历史缺陷库，用例集必须逐条体现覆盖（不涉及项须显式标"不涉及"并说明原因），否则 GATE 不退 QA。

## 八、历史缺陷模式库（已知逃逸点 → Few-shot）

> 下列为本项目已发生/已识别的逃逸模式。下次类似需求，Agent 须在用例中**主动设计对应负向用例**。

| 编号 | 缺陷模式 | 根因 | 触发检查单 | 必带用例 |
|------|---------|------|-----------|---------|
| ESC-001 | 班级切换弹窗只显"语文"，设置页却配了语文+数学 | 多数据源未对齐：ClassSwitchPage 用 mock，SettingsPage 用真实 `INIT` | #1 多租户/数据一致性、跨页链路 | 设置页配多科 → 切换页必须全显 |
| ESC-002 | 头部硬编码"语文·四年级(1班)"，切换后不更新 | AppLayout 头部未接 TeachingContext | #1、状态同步 | 切换班级 → 头部实时变 |
| ESC-003 | 顶部 `▾` 下拉纯装饰不可点 | 缺 onClick/下拉组件 | 核心交互 GATE-04 | 点击展开、可选切换 |
| ESC-004 | 首页用户卡 fallback 假名"张真真" | localStorage fallback 非真实数据 | #10 脱敏/真实源 | 改名后全站同步 |
| DEP-001 | Nginx `/ai/` 规则抢在 `/api/v1` 前匹配 | 路由顺序错误 | 部署变更生命周期 | 改 nginx 必须走 `deploy-log/` 各方确认闭环 |
| DEP-002 | 系统 nginx 占 80 端口、容器间连接拒绝 | listen_addresses 默认 localhost | 部署变更生命周期 | 端口/监听地址变更须登记 |
| AUTH-001 | cloud token 错误/过期未拦截 | 验签分支缺失 | #3 API 鉴权 | 错误/过期/重放 token 均 401 |

**反馈闭环使用**：线上每发现一个新逃逸缺陷，追加一行到本表，并回溯它本应在阶段一/二的哪条检查单被拦截；连续 2 次同类逃逸，则升级为 GATE 强制项。
